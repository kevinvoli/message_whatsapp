## Résumé exécutif

Le système QCM est ~80 % implémenté : infrastructure BDD (9 tables), backend (5 services, 2 contrôleurs, 28 endpoints) et admin UI (5/6 onglets) sont en place. Ce plan complète les manques en 5 phases, priorisées P0 → P2.

**Contrainte fondamentale** — un QCM programmé pour un jour donné ne peut être soumis **qu'une seule fois par jour par commercial**. Une fois soumis, le commercial n'est plus bloqué, **même après reconnexion**. Cette garantie repose sur le couple `(quiz_session.session_date = aujourd'hui, quiz_attempt.completed_at IS NOT NULL)` et doit être verrouillée côté backend (source de vérité) ET reflétée côté frontend (UX). Voir section « Idempotence QCM ».

**Effort estimé global** : Phase 1 (M), Phase 2 (S), Phase 3 (M), Phase 4 (L), Phase 5 (L).

### Points clés découverts pendant l'audit du code (au-delà du rapport)

1. `getTodaySession()` (`quiz-attempt.service.ts:99-103`) ne filtre **pas** `deletedAt` — bug confirmé.
2. La modale bloquante au login se branche dans `front/src/app/layout.tsx` via un wrapper, **exactement comme `IdleAndCooldownWrapper`** déjà en place (élément réutilisable majeur).
3. `quiz_session.session_date` a une contrainte **UNIQUE** → bloque structurellement le multi-QCM/jour. La Phase 5 doit lever cette contrainte par migration.
4. Le lien poste ↔ canal existe déjà : `WhapiChannel.poste_id` (mode canal dédié). Le ciblage Phase 4 doit s'appuyer dessus, pas réinventer.
5. `getSessionResults()` (`quiz-admin.service.ts:157-230`) est complet, zéro N+1 (2 requêtes agrégées) — Phase 2 = pur câblage UI.

## Risques de duplication

1. **`PdfDrawer` (quiz/page.tsx) vs page « Mes cours » (Phase 3)** — les deux listent les PDFs accessibles via `getQuizPdfs()` avec boutons Voir/Télécharger. **Action** : extraire `PdfList`/`PdfCard` dans `front/src/components/quiz/PdfList.tsx` **avant** d'écrire la page Mes cours, et faire pointer le `PdfDrawer` existant dessus. (Tâche 3.0)
2. **Tableau de résultats admin vs historique commercial** — formats proches (lignes score/réussite). Périmètres et données distincts (admin = tous les commerciaux ; commercial = soi-même) → **pas de factorisation forcée**, mais réutiliser les mêmes helpers de formatage score/badge `dateUtils`.
3. **Logique d'idempotence dupliquée backend/frontend** — risque que le frontend ré-implémente « déjà soumis aujourd'hui ». **Action** : le frontend ne calcule jamais l'état ; il consomme `attemptCompleted` renvoyé par `getTodaySession()`. Source unique = backend.
4. **Ciblage (Phase 4) vs exemption** — ne pas créer une logique de scope parallèle. Étendre le pattern `isExempt()` (résolution commercial+poste) dans le même service. (voir Décisions d'architecture)

### Décision de schéma : **nouvelle table `quiz_targeting`** (voir « Décisions d'architecture »)

### Tâches

**4.0 — Migration `quiz_targeting`** · backend · **M**
- Fichier (nouveau) : `message_whatsapp/src/database/migrations/AddQuizTargeting<timestamp>.ts`.
- Table `quiz_targeting` :
  - `id` UUID PK
  - `session_id` varchar(36) FK → `quiz_session(id)` ON DELETE CASCADE

## Décisions d'architecture

### D1 — Idempotence : garantie au niveau service, pas contrainte unique partielle
MySQL ne supporte pas les index uniques partiels (`WHERE completed_at IS NOT NULL`). Tenter un index unique sur `(commercial_id, session_id)` casserait les tentatives multiples légitimes (`maxAttempts > 1`). **Décision** : la garantie « 1 complétée/jour » est portée par le service (`startAttempt` rejette si une tentative complétée existe quand `maxAttempts === 1`), exécuté dans une transaction. La migration Phase 1 se limite à la FK d'intégrité commerciale. Ceci reste robuste car le backend est l'unique point d'écriture (pas de double client concurrent réaliste pour un même commercial ; en cas de course, la transaction + relecture `completedAt` tranche).

### D2 — Ciblage : nouvelle table `quiz_targeting` plutôt qu'extension de `quiz_exemption`
**Choix : table dédiée `quiz_targeting`.** Raisons :
- Sémantique opposée — `quiz_exemption` = exclusion globale (tous QCM), ciblage = inclusion **par session**. Mélanger un `direction: include|exclude` dans `quiz_exemption` surchargerait une table au scope global et compliquerait `isExempt()`.
- Le ciblage est lié à `session_id` (FK), l'exemption ne l'est pas.
- Ajout du scope `channel` (absent des exemptions) sans toucher au comportement existant.
- Le service `QuizTargetingService.isTargeted()` reste un miroir lisible de `isExempt()` → factorisation conceptuelle sans couplage.
**Alternative rejetée** : colonnes array `targeted_poste_ids`/`targeted_channel_ids` sur `quiz_session` — non requêtable proprement en SQL, viole la normalisation, complique les jointures de résolution.

### D3 — Obligation multi-QCM (Phase 5)
Trois options : (a) un seul obligatoire, (b) tous obligatoires séquentiels, (c) configurable par session (`is_mandatory`). **Recommandation : (c) avec défaut obligatoire**, en ajoutant `is_mandatory TINYINT DEFAULT 1` à `quiz_session` lors de la migration 5.1. Le wrapper ne bloque que sur les sessions `is_mandatory = 1` non soumises. Flexible sans surcoût.

### D4 — Modale bloquante : wrapper dans `layout.tsx`, pas dans la page WhatsApp
Monter `QuizGateWrapper` dans `layout.tsx` (comme `IdleAndCooldownWrapper`) garantit le blocage sur **toutes** les routes authentifiées (`/whatsapp`, `/contacts`, `/courses`), pas seulement la page conversations. Cohérent avec le pattern existant et non contournable par navigation directe d'URL.

## Idempotence QCM — garantie « 1 soumission / jour / commercial »

**Définition** : pour une session du jour (`session_date = CURDATE()`, `is_active = 1`, `deleted_at IS NULL`), un commercial non exempté et ciblé ne peut compléter qu'une tentative ; après soumission il n'est plus bloqué, y compris après reconnexion.

**Source de vérité unique = backend.** Le frontend ne calcule jamais l'état ; il consomme `getQuizToday()`.

**Mécanisme en couches :**

1. **Lecture (`getTodaySession`)** — après fix soft-delete (1.1), expose `alreadySubmittedToday = attempts.some(a => a.completedAt !== null)` (1.2). C'est l'unique signal lu par le wrapper.

2. **Écriture (`startAttempt` / `submitAttempt`)** — verrou (1.3) :
   - `startAttempt` rejette (`ForbiddenException`) si une tentative **complétée** existe déjà pour `(commercialId, sessionId)` et `maxAttempts === 1`.
   - `submitAttempt` rejette déjà si `attempt.completedAt !== null` (L.222) → pas de double soumission de la même tentative.
   - Exécution en transaction → en cas de course, relecture de `completedAt` tranche.

3. **Intégrité base (1.4)** — index existant `(commercial_id, session_id, attempt_number)` empêche les doublons de numéro de tentative ; FK ajoutée vers `whatsapp_commercial`.

4. **Frontend (1.5/1.6)** — `QuizGateWrapper` bloque ssi `sessionActive && !isExempt && !alreadySubmittedToday`. Après soumission, re-fetch/MAJ store (1.6) → `alreadySubmittedToday: true` → plus de blocage. Reconnexion → `getQuizToday()` relit l'état réel → cohérent sans persistance client.

**Pourquoi la reconnexion ne re-bloque pas** : aucun état n'est stocké côté client ; le blocage dérive entièrement de `getQuizToday()` qui lit `quiz_attempt.completed_at` en base. Une tentative complétée reste complétée → `alreadySubmittedToday` reste `true` toute la journée.

**Cas limites couverts :**
- Tentative démarrée mais non soumise (`completed_at IS NULL`) → toujours bloquant → `startAttempt` retourne la tentative en cours (L.170-180) plutôt que d'en créer une nouvelle. Correct.
- Exempté → `attemptCompleted: true` renvoyé directement → jamais bloqué.
- Changement de jour (minuit) → nouvelle `session_date` → nouveau cycle. **Vérifier la TZ MySQL** (point d'attention Phase 1).
